# Security Design and Threat Model

## 1. Trust boundaries

```text
Untrusted web page
   │ Content Script isolated world
Browser extension
   │ Token-authenticated WebSocket
Local Agent
   │ Project-root path restriction
Local project source
```

Page content, DOM attributes, and audit evidence are all treated as untrusted input.

## 2. Browser side

- Manifest V3; no remote JavaScript execution.
- The Side Panel CSP allows only bundled scripts and the local WebSocket.
- The page entry point uses Shadow DOM to reduce CSS contamination.
- The highlight layer does not receive pointer events.
- Temporary previews support only constrained attribute and style dictionaries.
- The extension does not call `eval` or `new Function` and does not execute code supplied by the page.

The current MVP uses `http://*/*` and `https://*/*` host permissions so an unpacked development build can audit pages without per-site configuration. A store release should instead:

1. Use `activeTab` by default.
2. Inject on demand with `chrome.scripting` after a user action.
3. Request persistent permission only for sites selected by the user.

## 3. WebSocket

- Binds to `127.0.0.1` by default and does not listen on LAN interfaces.
- The first message must be `hello`.
- The connection token is generated with a cryptographically secure random source, stored in `.nova/agent.json`, and written with file mode `0600`.
- Token comparison uses constant-time comparison.
- Chrome/Firefox extension origins and localhost development origins are allowed by default.
- The extension CSP permits connections only to localhost/127.0.0.1.

## 4. File safety

- The project root is canonicalized with `realpath` at startup.
- All file paths originate from the Agent's internal candidate search.
- Before every write, the path is resolved again and checked for project-root escape.
- Dependency, build-output, Git, and cache directories are excluded.
- The original file SHA-256 is recorded when a patch is generated.
- If the hash differs before apply, the Agent refuses to overwrite the file.
- If the post-patch hash differs before rollback, the Agent refuses to overwrite the file.

## 5. Command safety

Only the following scripts are allowed:

```text
typecheck
test
build
```

- Command names are not generated by the page or a language model.
- Execution uses `spawn(command, args, { shell: false })`.
- The timeout is 120 seconds.
- Captured output is size-limited.
- Custom arguments, pipes, redirection, and environment overrides are not supported.

## 6. Automatic patch limits

- Automatic patches use deterministic rules and never execute page HTML as code.
- Accessible names and alt text remain suggestions whose semantics require human review.
- Low-confidence mappings do not expose a “Confirm write” action.
- Every write must show a diff in the Side Panel and require an explicit user click.

## 7. Future hardening

- Replace the static token with a one-time pairing code and short-lived session keys.
- Add an extension-ID allowlist.
- Add WebSocket message-size and rate limits.
- Generate patches inside a Git Worktree.
- Add signed audit logs.
- Replace the local port with Native Messaging in the production release.
## 8. Animal interaction permission boundary

- The in-page 3D fox may emit only finite actions from the `NovaPetAction` allowlist.
- Audits, issue navigation, and temporary previews use deterministic handling inside the Content Script.
- Source-related actions must open the Side Panel through the Background worker and then run through an authenticated Local Agent.
- Applying a patch still requires an applicable diff, matching file hashes, and a backup; animal interaction cannot bypass these checks.
- The animal cannot supply shell strings, arbitrary file paths, remote scripts, or parameters outside the defined schemas.
- `pnpm check:pet-actions` verifies that every public action exists in the protocol, animal entry point, and corresponding handler.
