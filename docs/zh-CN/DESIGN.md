# YK-PETS Browser Agent 产品与交互设计

## 1. 产品定义

YK-PETS 是一个住在允许扩展运行网页右下角的 3D AI 前端工程伙伴平台。当前默认宠物为：

```text
名字：云灵（Zeph）
物种：云狐（Cloud Fox）
```

用户通过点击、双击、右键、悬停和拖拽云灵发起审计、问题导航、临时预览和工程操作；Side Panel 展示详细报告、Network Lab、Diff 与命令输出。连接本地 CLI 后，可在用户确认的前提下把建议映射到源码、生成最小补丁、执行检查并回滚。

产品核心链路：

```text
观察网页 → 解释问题 → 定位元素 → 临时预览 → 生成源码补丁 → 用户确认 → 验证 → 保留或回滚
```

## 2. 品牌和角色规则

- `YK-PETS` 只表示产品与平台；
- `云灵 / Zeph` 表示当前宠物名字；
- `云狐 / Cloud Fox` 表示当前宠物物种；
- 产品标题不得使用宠物名字替代品牌；
- 宠物对话、动作和角色介绍优先使用宠物名字；
- 物种特征、尾巴、触角和能力介绍使用物种名；
- 后续新增宠物不会改变 YK-PETS 品牌。

## 3. 目标用户

### 前端开发者

希望在开发 localhost 项目时快速发现可验证的问题，并在不离开浏览器的情况下定位源码、生成差异和运行项目脚本。

### 产品、设计与测试人员

希望对测试环境或线上页面进行可视化检查，快速指出具体元素并预览修改效果，不要求具备代码编辑能力。

### 技术负责人

希望把检查结果转化为可重复工程流程，并确保任何自动修改都可审阅、可验证、可回滚。

## 4. 产品表面

```text
网页右下角的云灵（Zeph）
├── 情绪、动作与语音反馈
├── 页面审计
├── 问题导航与定位
├── 临时预览与撤销
├── 详细报告入口
└── 本地工程工具箱

YK-PETS Side Panel
├── 当前页面与健康度
├── 指标和完整问题列表
├── 审计规则范围
├── Network Lab 与 Mock 规则
├── Local Agent 连接
├── 源码候选与补丁 Diff
├── 确认写入与回滚
└── Typecheck / Test / Build 输出
```

网页内宠物是自然交互入口；Side Panel 是详细信息与高风险确认区。

## 5. 宠物状态反馈

云灵不是装饰，而是系统状态的角色化表达：

| 系统状态 | 云灵表现 |
|---|---|
| 等待互动 | 靠近倾听、观察指针 |
| 正在审计 | 思考、核心脉冲 |
| 页面健康 | 开心、尾巴摇动 |
| 高优先级问题 | 疑惑、歪头 |
| 正在生成补丁 | 思考、环绕光加速 |
| 补丁已应用 | 兴奋庆祝 |
| 页面状态较差 | 趴下休息 |

所有关键状态同时通过可读文本表达，不能只依赖动画或颜色。

## 6. 主要交互

| 交互 | 结果 |
|---|---|
| 单击云灵 | 展开或收起功能菜单 |
| 双击云灵 | 立即执行页面审计 |
| 右键云灵 | 打开工程工具箱 |
| 悬停云灵 | 打招呼并跟随指针 |
| 拖拽云灵 | 调整网页内位置 |
| 点击问题徽标 | 进入问题导航 |

双击和右键是快捷方式，所有操作必须同时提供键盘可访问入口。

## 8. 文案原则

- 不使用“自动修好了”之类绝对表述；
- 明确区分“网页临时预览”和“项目源码修改”；
- 映射不可靠时展示候选和原因，不伪造确定性；
- 失败文案说明下一步；
- 角色说话使用“我”，工程确认区使用明确的 YK-PETS 产品名称；
- 不再使用 NOVA 作为当前品牌或宠物名字；
- 历史 NOVA 只允许出现在版本历史和兼容说明中。

## 9. 无障碍与性能

- 所有操作必须可键盘访问；
- 图标按钮必须有文本或 `aria-label`；
- 健康度和严重级不能只靠颜色；
- 宿主页面高亮层不能阻断页面交互；
- Shadow DOM 隔离宿主样式；
- 标签页隐藏时应暂停 WebGL 渲染；
- `prefers-reduced-motion` 下减少动作幅度；
- 低性能设备和 WebGL 不可用时提供 2D 回退。

## 10. 多宠物设计约束

未来每个宠物都必须通过独立身份和能力声明接入：

- 名字和物种分别本地化；
- 通用意图映射到宠物自己的动作；
- 不支持的动作有明确降级；
- 宠物资源按需加载；
- 切换宠物不改变审计、网络或 Agent 业务状态；
- 用户数据以稳定宠物 ID 存储，不能依赖显示名。

第二只宠物是平台化验收条件，用于证明架构不再只适配云狐。
