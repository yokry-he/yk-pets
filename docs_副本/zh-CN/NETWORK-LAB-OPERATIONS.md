# 网络实验室与 Mock 工作台详细操作

> 本文专门描述网络拦截、Mock、响应修改、规则冲突和性能分析。普通用户可先阅读 [使用操作手册](./USER-GUIDE.md)。

## 1. 执行范围

当前版本拦截页面主世界中的 `fetch` 和 `XMLHttpRequest`。静态脚本、样式、图片、字体和媒体资源通过 Performance Resource Timing 采集，但不修改正文。

暂不覆盖：

- WebSocket 帧；
- Server-Sent Events 流；
- Service Worker 内部发起且不经过页面 Hook 的请求；
- 流式响应的逐块修改；
- 浏览器内部页面。

## 2. 总开关和采集开关

- **总开关**：控制 Mock、响应修改和延迟。
- **采集开关**：控制请求与资源性能记录。

建议：

- 开发 Mock 时同时开启两者；
- 只做真实性能分析时关闭总开关、保留采集；
- 完成测试后关闭总开关，防止忘记 Mock 状态。

## 3. 完整 Mock 的执行顺序

```text
页面调用 fetch/XHR
→ 读取当前 Origin 设置
→ 对启用规则排序
→ 匹配 URL、方法、Query 和 Scope
→ 命中 mock 规则
→ 等待配置延迟
→ 构造 Response 或模拟 XHR 完成事件
→ 记录 Mock 请求
→ 不发送真实网络请求
```

## 4. 不存在接口的示例

页面代码：

```ts
const result = await fetch('/api/future-feature').then(response => response.json())
```

规则：

```text
Scope: https://app.example.com
Method: GET
Pattern: /api/future-feature
Action: Mock
Status: 200
```

响应：

```json
{
  "enabled": true,
  "items": ["alpha", "beta"]
}
```

启用规则后，页面不需要后端服务即可收到该 JSON。

## 5. 从请求生成 Mock

“生成 Mock”使用捕获到的响应预览。预览可能因安全限制而截断，因此保存前需要检查：

- 数组是否完整；
- 长字符串是否截断；
- 敏感字段是否已被替换；
- Content-Type 是否符合页面预期；
- 状态码是否需要改为测试场景状态。

需要精确回放大响应时，后续应使用 HAR/录制回放能力，而不是依赖当前有限预览。

## 6. 规则测试

测试区域至少填写：

- 测试 URL；
- HTTP 方法。

测试结果显示：

- 当前候选规则是否命中；
- 最终会被选中的规则；
- 同时命中的竞争规则；
- 优先级和精确度导致的排序。

保存前必须处理危险级冲突。

## 7. 优先级和精确度

推荐优先级约定：

| 范围 | 建议优先级 |
|---|---:|
| 全站兜底 | 10–30 |
| 业务模块通用规则 | 50–80 |
| 单接口规则 | 100 |
| 临时调试覆盖 | 150–200 |

匹配越具体，精确度越高。避免两个相同 URL、方法、Query 和优先级的规则同时启用。

## 8. 响应修改

修改真实响应适合以下场景：

- 强制开启隐藏功能；
- 注入测试权限；
- 删除内部字段；
- 修改分页总数；
- 制造空状态或异常字段。

不要用它修改非 JSON、二进制或流式响应。

## 9. 性能数据理解

### TTFB

从请求开始到收到首字节。占比高通常表明服务端处理、网络往返或代理链路较慢。

### Download

从首字节到响应完成。耗时高通常与响应体积、带宽、压缩或流量整形有关。

### Transfer Size

网络传输体积。跨域服务器未配置 Timing-Allow-Origin 时，浏览器可能隐藏部分 Timing 或大小字段。

### P95

将请求按耗时排序后的第 95 百分位，用于避免平均值掩盖长尾请求。

## 10. 用 Mock 定位性能瓶颈

1. 记录真实接口耗时和页面可用时间。
2. 为慢接口创建低延迟 Mock。
3. 刷新并重复相同操作。
4. 对比：
   - 页面大幅变快：网络或服务端是主要瓶颈；
   - 页面仍然慢：检查 JSON 解析、状态计算、DOM 渲染和大列表；
   - 缩小 Mock 数据后变快：检查数据量和渲染复杂度。

## 11. 安全操作清单

- 确认顶部 Origin；
- 使用最小匹配范围；
- 复制规则默认保持关闭；
- 不在响应中保存真实 Token；
- 分享规则前执行脱敏；
- 完成测试后关闭总开关；
- 发布构建前确认默认设置为关闭。

## v0.6.0 请求逐条展开与规则同步

### 展开请求详情

“请求”Tab 中的每一条记录都提供“展开详情”按钮。展开后可查看：

- 完整 URL 和 Query 参数；
- 状态码、耗时、体积和请求来源；
- DNS、连接、TLS、TTFB 和下载时间；
- 请求头和响应头；
- 请求体预览和响应预览；
- Mock、响应修改以及错误状态。

`Authorization`、Cookie、Token、密码、Secret、Session 和 Credential 等敏感内容会在进入 Side Panel 前脱敏。

### 基于请求生成规则

展开请求后点击“基于请求生成规则”，系统会打开全页规则编辑器并带入 URL、方法、状态码和可用响应预览。保存后规则同时写入 Chrome Storage，并立即同步到当前页面拦截器。

### 直接创建不存在的接口

进入“规则”Tab 后点击“新增规则”，手动填写任意相对路径或完整 URL。保存并启用后，只要页面后续发起匹配的 Fetch/XHR，请求会直接返回 Mock 数据，不要求真实服务器存在该接口。
