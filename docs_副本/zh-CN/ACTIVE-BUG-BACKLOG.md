# NOVA 当前缺陷清单

> 当前状态：六项问题已在 `v0.6.7` 完成实现，等待最终浏览器安装回归确认。

## BUG-001：打开具体 Side Panel 页面后加载环永久显示

- 状态：已完成（`v0.6.7`）
- 优先级：P1
- 发现版本：`v0.6.6`
- 影响范围：网页内 3D 宠物、Side Panel 导航、任务状态反馈

### 复现步骤

1. 当前没有页面审计、补丁生成、项目验证或 Agent 请求正在执行。
2. 通过宠物右侧功能进入某个具体 Side Panel 页面，已确认“网络实验室”可以复现。
3. Side Panel 已正常打开，Network Lab 总开关也可以处于关闭状态。
4. 宠物头部上方出现加载环，并持续存在，不会自动消失。

### 实际结果

普通页面导航被表现为正在执行任务。即使没有任务，宠物仍保持加载/思考视觉状态。

### 预期结果

- 打开完整报告、网络实验室或 Agent 设置页只属于导航，不应进入忙碌状态。
- 加载环只能在真实异步任务进行时显示。
- 任务成功、失败、取消或 Side Panel 未接收动作时都必须可靠复位。

### 初步定位

当前仅记录定位结果，不实施修改：

1. 页面 Content Script 将 `network-lab` 当作委托任务，写入 `behavior: thinking` 和 `busy: true`。
2. Side Panel 打开成功后，页面侧没有对应的成功复位逻辑。
3. Side Panel 消费 `network-lab` 动作时只切换工作区并返回，没有明确同步 `busy: false`。
4. 3D 模型的数据光环在 `behavior === thinking` 或任务执行状态下都会显示，因此残留的 `thinking` 也会造成永久加载环。

### 统一修改时的建议方案

- 区分“页面导航动作”和“真实任务动作”。
- 报告、网络实验室和设置页导航始终保持 `busy: false`。
- Side Panel 消费导航动作后返回明确的完成状态。
- 加载环只由真实任务执行状态控制，不由普通菜单视觉动作控制。
- 为跨 Context 任务增加完成、失败和超时复位保障。

### 验收标准

- 打开网络实验室不会出现持续加载环。
- 打开报告或 Agent 设置不会错误进入忙碌状态。
- 生成补丁、运行检查等真实任务仍会显示加载状态。
- 真实任务在成功、失败和超时后都能关闭加载状态。
- 重复开关 Side Panel、切换工作区和刷新页面后状态保持一致。

## BUG-002：尾巴尖在深色背景下不可见

- 状态：已完成（`v0.6.7`）
- 优先级：P2
- 发现版本：`v0.6.6`
- 影响范围：3D 宠物材质、尾巴辨识度、浅色与深色网页适配

### 复现步骤

1. 在浅色网页或深色网页中显示 3D 宠物。
2. 打开宠物菜单，使尾巴和尾尖完整可见。
3. 对比浅色背景与深色背景下的尾尖颜色和轮廓。
4. 在深色背景下，尾尖当前颜色与周围光效混合，轮廓接近不可见。

### 实际结果

尾尖的基础颜色和亮度不足。当前发光主要依赖动作或能量状态，普通待机时缺少稳定的明亮核心，因此在深色背景中无法清楚识别尾巴末端。

### 预期结果

- 尾尖使用比当前稍亮的青绿色或薄荷色。
- 尾尖在待机、菜单打开和普通动作期间始终保持基础发光。
- 特殊动作可以在基础发光之上增强亮度和脉冲，但不能让普通状态完全熄灭。
- 浅色背景下不应过曝，深色背景下应有清晰轮廓。

### 统一修改时的建议方案

- 为尾尖设置常驻的最低自发光强度。
- 使用“明亮实体核心 + 柔和透明外层光晕”的双层表现。
- 选择比当前更亮、偏青白的颜色，同时保持现有云狐配色关系。
- 将现有动作驱动亮度改为基础值之上的增量，而不是从接近零开始。
- 分别在白色、浅灰、深灰和纯黑背景下校验对比度。

### 验收标准

- 深色背景中可以立即辨认尾尖的位置和轮廓。
- 待机状态下尾尖持续发光，不闪烁、不间歇消失。
- 动作发光仍有明显强弱变化，不会因为常亮而失去反馈层级。
- 浅色背景下尾尖不过曝，不遮盖尾巴本体连接关系。
- Content Script 和 Playground 中的宠物表现一致。

## BUG-003：动作思考框尺寸过大并遮挡页面操作

- 状态：已完成（`v0.6.7`）
- 优先级：P1
- 发现版本：`v0.6.6`
- 影响范围：动作反馈、菜单操作、页面内容可用性

### 复现步骤

1. 打开宠物动作菜单。
2. 选择“吃饭”等带文案反馈的动作。
3. 页面左上方出现接近通知面板尺寸的大型提示框。
4. 提示框覆盖菜单、页面内容或可点击区域，并提供最小化和关闭操作。

### 实际结果

动作对应文案使用了大型任务/通知面板呈现。该面板占用空间过多，会遮挡页面内容并影响后续操作，不符合轻量“思考气泡”的预期。

### 预期结果

- 动作文案显示为宠物头部附近的小型思考气泡。
- 气泡只展示动作标题和一至两行短文案。
- 气泡不提供最小化、关闭或其他按钮。
- 气泡不能拦截鼠标事件，也不能遮挡环形菜单的主要操作区。
- 气泡在短时间后自动淡出；如果无法稳定避免遮挡，则可以完全不显示。

### 统一修改时的建议方案

- 将“动作反馈气泡”与“审计/任务通知面板”彻底拆分为两个组件和状态通道。
- 小气泡锚定宠物头部，根据窗口边界自动向左或向右避让。
- 使用较小最大宽度、两行截断、短时间淡入淡出和 `pointer-events: none`。
- 动作切换时直接更新或替换当前气泡，禁止堆叠多个提示。
- 保留大型面板仅用于需要用户阅读或操作的审计、错误和任务结果。

### 验收标准

- 触发动作后不会再出现截图中的大型提示框。
- 小气泡不遮挡动作菜单和页面主要可点击元素。
- 气泡不可点击、不可聚焦，并会自动消失。
- 连续切换动作不会产生多个重叠气泡。
- 在窄屏和页面边缘位置无法安全显示时，允许隐藏气泡。

## BUG-004：动作音频是提示音而不是对应文案语音

- 状态：已完成（`v0.6.7`）
- 优先级：P1
- 发现版本：`v0.6.6`
- 影响范围：动作反馈、声音设计、后续多语言能力

### 实际结果

动作触发时使用 Web Audio 合成的“滴”声或音高变化作为反馈，与动作内容和屏幕文案没有语义关系。

### 预期结果

- 每个保留动作播放与该动作气泡文案对应的语音。
- 声线可爱、轻柔，符合数字宠物形象，而不是系统通知音。
- 同一动作的文案和语音内容保持一致。
- 重复点击当前动作不重复播放语音。
- 新动作打断旧动作时，旧语音停止，不能重叠播放。
- 如果对应语音无法播放，应保持静音，不使用“滴”声回退。

### 统一修改时的候选方案

实施前需要确定语音来源：

1. 内置预生成语音资源：声音一致、可离线使用，但会增加安装包体积，需要确认生成方式和资源授权。
2. 浏览器 `speechSynthesis`：无需额外插件，但不同系统的声线差异较大，无法保证“可爱”效果。
3. 可选 TTS Provider：声音质量和多语言扩展更好，但需要网络、密钥和 Provider Adapter。

如果最终方案需要外部语音生成插件或服务，必须先获得用户确认并完成插件安装，再生成或接入语音资源。

### 建议默认策略

- 手动触发动作时播放语音。
- 自动闲时动作默认静音，避免打扰页面阅读。
- 提供全局静音和音量设置，并遵守浏览器自动播放限制。
- 文案使用统一语言键，为后续多语言语音包预留映射。

### 验收标准

- “吃饭”播放与吃饭文案一致的可爱语音，不再播放提示音。
- 所有保留动作都有明确的文案—语音映射。
- 新动作触发时不会与旧语音重叠。
- 静音、播放失败和浏览器限制情况下不会出现“滴”声回退。
- 不安装或调用未经用户确认的外部插件与语音服务。

## BUG-005：请求列表缺少按资源类型分类和筛选

- 状态：已完成（`v0.6.7`）
- 优先级：P1
- 发现版本：`v0.6.6`
- 影响范围：Network Lab 请求列表、网络分析效率、筛选体验

### 当前结果

“请求”页只提供 URL、方法或状态码文本筛选，以及“只看慢请求”开关。大量请求混在同一列表中，无法像浏览器 DevTools Network 面板一样快速查看接口、脚本、样式、图片或 WebSocket 等不同资源。

### 预期结果

- 在请求搜索栏附近增加资源类型快捷筛选栏。
- 首批类型与 DevTools Network 的常用分类保持接近：`全部`、`Fetch/XHR`、`文档`、`CSS`、`JS`、`字体`、`图片`、`媒体`、`Manifest`、`Socket`、`Wasm`、`其他`。
- 每个筛选项可以显示当前已采集请求中的数量，数量随请求采集和清空操作实时更新。
- 选择资源类型只过滤当前列表，不改变“请求总数”和底层采集数据。
- 类型筛选可以与现有文本搜索及“只看慢请求”同时生效。
- 请求详情中显示最终识别出的资源类型，便于核对分类结果。

### 统一修改时的建议方案

- 在请求采集模型中增加稳定的 `resourceType` 字段，不在渲染阶段仅靠文件扩展名临时猜测。
- 优先使用浏览器提供的请求类型；对 Fetch、XHR、WebSocket 和 Performance API 数据进行归一化。
- 浏览器未提供明确类型时，依次参考请求来源、`Content-Type`、URL 后缀和响应头，并最终归入“其他”。
- 统一定义类型枚举和中文/英文显示映射，避免 Content Script、后台采集和 Side Panel 各自维护不同规则。
- 分类栏支持横向滚动或自动换行，不能挤压搜索框或造成 Side Panel 横向溢出。
- 保留后续增加“按方法”“按状态”“被拦截/已修改”等二级分类的扩展能力。

### 验收标准

- Fetch 与 XHR 请求均进入 `Fetch/XHR` 分类。
- HTML 文档、CSS、JavaScript、字体、图片、音视频、Manifest、WebSocket 和 Wasm 可以正确归类。
- 无法识别的请求稳定进入“其他”，不会从列表消失。
- 类型筛选、文本筛选和慢请求筛选组合后的结果正确。
- 新请求到达、切换页面和清空记录后，各分类数量立即同步。
- 300 条以上请求时切换分类保持流畅，且不会重复采集或丢失请求。

## BUG-006：页面审计缺少问题分类与审计范围控制

- 状态：已完成（`v0.6.7`）
- 优先级：P1
- 发现版本：`v0.6.6`
- 影响范围：页面审计配置、审计执行流程、报告评分与问题列表

### 当前结果

页面审计会固定执行全部规则。审计完成后只能按高、中、低严重程度筛选结果，用户无法按问题类型查看结果，也无法在执行前关闭不需要的审计类型。

### 预期结果

- 在“开始页面审计”附近增加“审计范围”多选配置。
- 首批分类沿用当前审计领域模型：`性能`、`无障碍`、`SEO`、`最佳实践`、`DOM / 页面结构`。
- 每个分类使用独立多选框控制，默认全部勾选。
- 未勾选的分类必须从审计执行流程中跳过，不运行对应规则，也不产生对应问题。
- 审计报告的问题列表增加分类筛选，并可继续与高、中、低严重程度筛选组合使用。
- 报告中明确显示本次实际启用的审计范围，避免把“未检查”误解为“没有问题”。

### 统一修改时的建议方案

- 为审计请求增加 `enabledCategories`，由 Side Panel 和宠物快捷审计入口传入 Content Script。
- 按分类注册审计规则，并在调用规则前判断分类是否启用，而不是先执行全部规则再过滤结果。
- 页面指标只采集生成基础报告所必需的数据；分类专属的重型采集在未勾选时一并跳过。
- 评分、问题总数及高/中/低汇总仅依据本次已启用分类计算。
- 在 `AuditReport` 中保存本次审计范围，确保报告、问题导航、重新审计和 Local Agent 补丁上下文一致。
- 审计配置保留用户最近一次选择；提供“全选”和“恢复默认”快捷操作。
- 如果全部分类均未勾选，禁用“开始页面审计”并提示至少选择一个类型。
- 宠物双击快捷审计使用最近保存的范围；首次使用时采用全选。

### 验收标准

- 取消勾选“性能”后，不执行性能规则，结果中不出现性能问题，评分也不包含性能扣分。
- 只勾选“无障碍”时，仅执行并展示无障碍规则与问题。
- 重新勾选分类后，下一次审计会恢复对应检查。
- 分类筛选与严重程度筛选可以正确组合，且只影响结果展示，不改变已生成报告。
- 报告始终显示“已检查”和“未检查”的分类，未检查类型不会被表达为零问题或已通过。
- Side Panel 按钮审计、宠物双击审计和重新审计使用一致的范围配置。
- 全部取消勾选时不会启动空审计，也不会进入无法结束的忙碌状态。

## v0.6.9 音频热修复

- `BUG-011`：v0.6.8 的18个 MP3 容器可解码，但实际波形峰值仅 `-91 dB`，属于静音文件。根因是离线合成器使用了错误的中文语音标识。
- 已使用有效的中文声线重新生成全部文案，并进行轻微升调、响度标准化和高低频清理。
- 新增音频内容检查：逐条验证解码、时长、平均音量与峰值，阻止静音文件再次进入发布包。

`BUG-011` 已进入 `v0.6.9`。后续新问题从 `BUG-012` 开始登记。

## v0.6.10 多音色结果

- `BUG-012`：单一离线电子声不适合作为所有用户的默认审美。保留该素材并正式定义为“星云外星人”，同时新增“萌系少女”“萌宠伙伴”和“静音”。
- 萌系音色使用 Chrome/操作系统已安装的中文 TTS 声线，不再使用上一轮离线机器人声硬加工；后台优先选择名称包含自然、神经或常见中文女声标识的声线。
- Side Panel 支持持久化选择和即时试听；页面宠物会在设置变化后立即切换，动作文案保持一一对应。
- 不直接复制任何具体动漫角色声线；“萌宠伙伴”只采用通用动漫宠物风格的音高和语速参数。

`BUG-012` 已进入 `v0.6.10`。后续新问题从 `BUG-013` 开始登记。
